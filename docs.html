Every client gets the same onboarding: a firm slug, a dashboard password, and an API key. What you do with them is up to you — plug directly into your own product, or use the dashboard and hosted verification page we already built. Nothing stops you from using both for different parts of your flow.

A

API Only

Your backend talks to our API directly — no dashboard, no hosted page, full control over your own UI.

CredentialUsed asPurpose
Firm slugX-Client-Id headerIdentifies which firm a request belongs to

Submit a verification directly

POST/api/v1/verify
curl -X POST https://18.185.59.156/api/v1/verify \
  --cacert aws_admin_cert.pem \
  -H "X-Client-Id: your-firm-slug" \
  -H "X-Api-Key: YOUR_API_KEY" \
  -F "country=NG" \
  -F "doc_type=passport" \
  -F "mode=3" \
  -F "user_ref=your_internal_applicant_id" \
  -F "id_image=@id_front.jpg" \
  -F "liveness_frames=@frame1.jpg" \
  -F "liveness_frames=@frame2.jpg"

Every submission is automatically tagged to your firm — results show up filtered to only your data if you ever log into the dashboard to review history, even though you're not required to use it day-to-day.

Optional — generate a link anyway, programmatically

If part of your flow still wants to hand off to our hosted capture page (e.g. you don't want to build the camera/liveness UI yourself, but everything else is your own backend), you can generate a one-time link with the same credentials — no dashboard login needed:

POST/api/v1/admin/sessions
curl -X POST https://18.185.59.156/api/v1/admin/sessions \
  --cacert aws_admin_cert.pem \
  -H "X-Client-Id: your-firm-slug" \
  -H "X-Api-Key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"user_ref": "your_internal_applicant_id"}'
{
  "ok": true,
  "token": "CjKt-NRk8rScP53wWBJKYDlzgPyGZdsM",
  "url": "https://id-verification-riea.onrender.com/index.html?token=CjKt-...",
  "expires_at": "2026-07-11T10:06:09.630214Z"
}

Redirect your user to url. It's single-use and expires in 24 hours — see Path B for what happens next.

Why a separate token instead of just using your API key for the link? Your API key is a long-lived secret — putting it in a URL a customer's browser holds means anyone who intercepts that link has standing access to your account forever. A session token is random, disposable, tied to one applicant, and dies the instant it's used. Worst case if one leaks: someone burns one verification slot. Your real key is never exposed.

B

Dashboard + Hosted Page

No code. Log in, generate a link, send it — the result lands back in your dashboard automatically.

What you're given at onboarding

  • A firm slug — your identifier on the login page
  • A dashboard password — set at onboarding, changeable any time

How it works, step by step

  1. Log in at the admin login page. Enter your firm slug in the "Firm" field and your password — you'll land on your Verifications list, scoped to only your firm's data.
  2. Click "Generate Link" in the nav, optionally add an applicant reference (your own internal ID for this person), and click Generate.
  3. Send the link you get back to your customer — email, SMS, chat, however you reach them.
  4. Your customer opens it on their own phone or laptop, and completes the document + liveness capture on our hosted page — no account, no app install.
  5. The link stops working the instant they finish (or after 24 hours, whichever comes first) — they can't reopen it, and neither can anyone else who might have seen it.
  6. The result appears in your dashboard, automatically tagged to your firm, alongside every other verification you've run.

This is the fix for "my client re-verified themselves twice with the same link" — once a link is used, it's dead. There is no way to reuse it, and no separate "normal URL" that bypasses this — a verification link is required to reach the capture flow at all.

Automating link generation without touching the dashboard

If your own site or app has a signup flow and you'd rather not have a human click "Generate Link" for every customer, your backend can call the exact same endpoint the dashboard button uses — see Path A's POST /sessions example above, using your firm's API key instead of a dashboard login. Same one-time link, same automatic dashboard visibility, zero manual steps.
